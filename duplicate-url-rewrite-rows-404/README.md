# Duplicate or orphaned url_rewrite rows cause 404s

Bulk product and category imports, Magento 1 to 2 migrations, and CSV re-imports without full category data routinely leave the `url_rewrite` table inconsistent, because Magento 2 rejects true duplicates instead of auto-suffixing them the way Magento 1 did, and removing or renaming an entity does not always retire its old autogenerated row. This script fetches the live product and category ids over REST, reads a read-only export of `url_rewrite`, flags duplicate and orphaned rows, and repairs the fixable ones by resaving the owning product with its own unchanged `url_key`.

**Full guide with diagrams:** https://www.allanninal.dev/magento/duplicate-url-rewrite-rows-404/

## Run it

`url_rewrite` has no public REST endpoint, so this script never writes to it directly. You need a read-only export of the table (CSV columns: `url_rewrite_id, entity_type, entity_id, request_path, target_path, redirect_type, store_id, is_autogenerated`) from an Admin grid export or a read-only database query.

```bash
export MAGENTO_URL="https://your-store.example.com"
export MAGENTO_ADMIN_TOKEN="your admin bearer token"
export URL_REWRITE_EXPORT_CSV="url_rewrite_export.csv"
export DRY_RUN="true"

python duplicate-url-rewrite-rows-404/python/url_rewrite_check.py
node   duplicate-url-rewrite-rows-404/node/url-rewrite-check.js
```

`classify_url_rewrites` is a pure function: it groups the rewrite rows by `(request_path, store_id)` to find duplicates and separately filters rows whose `entity_id` is missing from the live product and category ids to find orphans. Duplicates and orphans that still point at a live product are repaired by a `PUT` to that product with its own unchanged `url_key`, which makes Magento regenerate and supersede its rewrite row. True orphans with no live entity at all are only reported, since deleting them needs CLI or direct database access plus a reindex. Start with `DRY_RUN=true` to review the report first.
