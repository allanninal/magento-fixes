# Duplicate url_key blocks product resave after migration

Magento 1 and many external carts or PIMs never enforced `url_key` uniqueness, so a migrated catalog frequently carries two or more products whose `url_key` collides once the store's URL suffix and category path are applied. Magento 2 enforces a unique constraint on `url_rewrite(request_path, store_id)`, so resaving one of the colliding products calls `UrlPersist`, which tries to insert a `request_path` row that already belongs to a different `entity_id` and throws an `AlreadyExistsException` ("URL key for specified store already exists"), blocking the save.

This script fetches every product over `GET /rest/V1/products`, groups them by their expected `request_path`, and for any group with more than one product keeps the row Magento already treats as authoritative (a manual redirect over an autogenerated one, else the lowest `entity_id`), reporting a deterministic new `url_key` of `sku-original_url_key` for every other colliding product. It only calls the repair `PUT` when `DRY_RUN=false`, because renaming a live `url_key` changes a public-facing SEO URL.

**Full guide with diagrams:** https://www.allanninal.dev/magento/duplicate-url-key-blocks-resave/

## Run it

```bash
export MAGENTO_URL="https://your-store.example.com"
export MAGENTO_ADMIN_TOKEN="your admin bearer token"
export URL_SUFFIX=".html"
export DRY_RUN="true"

python duplicate-url-key-blocks-resave/python/duplicate_url_key.py
node   duplicate-url-key-blocks-resave/node/duplicate-url-key.js
```

`resolve_url_key_collisions` / `resolveUrlKeyCollisions` is a pure function: it groups products by their expected `request_path` and `store_id`, and for any group spanning more than one `entity_id` it picks a keeper (preferring a manual redirect over an autogenerated row, else the lowest `entityId`), and derives a new `url_key` for every other colliding product. It does no I/O, so it needs no network, no database, and no Magento store to test. The HTTP calls both live behind `run()`, so importing the module never requires credentials. Start with `DRY_RUN=true` to review the planned renames first, and never rename a `url_key` a script has not confirmed collides with another live product. Truncating or deleting rows directly in `url_rewrite` is not done via REST and is not something this script attempts; that path needs CLI access, a full reindex with `bin/magento indexer:reindex catalog_url_rewrite_product`, cron, and a cache flush.
